# Галереї для фотографів — дизайн структури

> Крок 1: структура сторінок, даних і потоків. Реалізація — окремими фазами (нижче).

## Ідея

Фотограф співпрацює з TouchMemories: отримує **сторінку-візитку** й може створювати
**галереї** з фото своїх клієнтів. Клієнт отримує посилання на галерею, дивиться і
завантажує фото протягом **30 днів**, після чого файли автоматично видаляються.
У галереї — CTA «Створити фотокнигу з цих фото» → конструктор TouchMemories
(взаємна вигода: фотографу — сервіс доставки фото, нам — трафік у конструктор).

## Сторінки

### 1. Візитка фотографа — `/{locale}/photographer/{slug}` (публічна)
```
┌──────────────────────────────────────┐
│  [логотип]     Ім'я Прізвище          │
│                Фотограф • Київ        │
│  ─────────────────────────────────    │
│  Про себе (2–3 речення)               │
│  📞 телефон   ✉ email                 │
│  📷 Instagram   🌐 сайт               │
│  ─────────────────────────────────    │
│  [Замовити фотокнигу зі своїх фото]   │  ← CTA у конструктор
└──────────────────────────────────────┘
```
- Slug — людський (`/photographer/olena-kovalenko`), задається в адмінці.
- Жодних приватних даних: галереї на візитці НЕ списуються.

### 2. Кабінет фотографа — `/{locale}/photographer/cabinet/{token}` (за токеном)
За зразком кабінету партнера (`/partner/{token}`): без реєстрації, доступ = UUID-токен з листа.
- **Профіль**: ім'я, біо, контакти, завантаження логотипа/аватара → так виглядатиме візитка й шапка галерей.
- **Галереї**: список із лічильником днів до видалення; кнопка «Нова галерея»
  (назва, ім'я клієнта, дата зйомки) → створюється посилання для клієнта.
- **Завантаження фото** у галерею: множинний вибір, прогрес, ліміти
  (тільки зображення, ≤ 25 МБ/файл, ≤ 500 фото/галерею).
- Кнопка «Скопіювати посилання для клієнта».

### 3. Галерея клієнта — `/{locale}/gallery/{token}` (за токеном)
```
┌──────────────────────────────────────┐
│  [лого] Ім'я фотографа   📞 ✉ 📷      │  ← блок-візитка (бренд фотографа)
│  «Весілля Олена & Максим» • 12.07     │
│  ⏳ Доступно ще 23 дні                │
│  ─────────────────────────────────    │
│  [Завантажити все (ZIP)]              │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐          │
│  │фото│ │фото│ │фото│ │фото│  … грід  │
│  └────┘ └────┘ └────┘ └────┘          │
│  ─────────────────────────────────    │
│  [Створити фотокнигу з цих фото] CTA  │
└──────────────────────────────────────┘
```
- Лайтбокс на клік, завантаження поштучно і ZIP (патерн уже є в адмінці замовлення).
- Після закінчення терміну: сторінка лишається, але замість гріда —
  «Термін зберігання минув» + контакти фотографа.

### 4. Адмінка — `/admin/photographers`
- Створення/редагування фотографа (slug, email), вмикання/вимикання.
- Кнопка «Надіслати лист із кабінетом» (Brevo, як у партнерів).
- Перегляд галерей фотографа й займаного місця.

## Дані (Supabase)

```sql
photographers (
  id uuid pk, slug text unique, cabinet_token uuid unique default gen_random_uuid(),
  name text, bio text, email text, phone text, instagram text, website text,
  logo_url text, avatar_url text,
  is_active bool default true, created_at timestamptz default now()
)

photographer_galleries (
  id uuid pk, photographer_id uuid fk,
  client_token uuid unique default gen_random_uuid(),   -- посилання клієнта
  title text, client_name text, shoot_date date,
  expires_at timestamptz default now() + interval '30 days',
  files_purged_at timestamptz,                          -- маркер очищення (як в orders)
  created_at timestamptz default now()
)

photographer_gallery_photos (
  id uuid pk, gallery_id uuid fk,
  storage_path text, file_name text, size_bytes bigint, sort_order int,
  created_at timestamptz default now()
)
```

## Сховище

- Новий бакет **`photographer-galleries`**, шлях `{photographer_id}/{gallery_id}/{file}`.
- **Усі записи тільки через service-role ендпоінт** (правило проєкту: браузер не пише
  в бакети напряму — уникаємо тих самих RLS-граблів, що були з order-files).
- Читання: публічні URL з невгадуваними шляхами (як в інших бакетах проєкту).

## API

| Роут | Доступ | Призначення |
|---|---|---|
| `GET/POST /api/photographers/profile` | cabinet_token | читання/оновлення профілю, логотип |
| `GET/POST /api/photographers/galleries` | cabinet_token | список/створення галерей |
| `POST/DELETE /api/photographers/galleries/{id}/photos` | cabinet_token | аплоад/видалення фото (service-role) |
| `GET /api/gallery/{client_token}` | публічний | дані галереї для клієнта |
| `GET /api/gallery/{client_token}/zip` | публічний | ZIP усіх фото |
| `CRUD /api/admin/photographers` | staff | адмінка |
| `GET /api/cron/cleanup-galleries` | CRON_SECRET | очищення прострочених |

## Ретеншн (30 днів)

- Нічний крон `cleanup-galleries` (додати у `vercel.json`): галереї з
  `expires_at < now()` і `files_purged_at is null` → видалити файли з бакета
  батчами, проставити `files_purged_at`. Рядки галереї/фотографа не чіпаємо.
- (Фаза 2) Лист клієнту й фотографу за 3 дні до видалення (Brevo).

## Безпека

- Кабінет і галерея — UUID-токени (невгадувані), без акаунтів: як у партнерів.
- Аплоад валідовано на сервері: токен → власник галереї, тип файлу (image/*),
  розмір ≤ 25 МБ, ліміт 500 фото/галерею.
- Прострочена галерея не віддає фото навіть до фізичного очищення.
